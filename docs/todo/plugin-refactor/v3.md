# Plugin System v3 — Third-Party Plugin Support

## Problem

The current plugin registry (v2) solves code organization: each plugin lives entirely
in `src/plugins/<name>/` and registers itself in `src/plugins/index.ts`. However,
npm dependencies are still a build-time concern. A third-party developer who wants to
ship a plugin still has to fork the repo, add packages to the root `package.json`,
run `bun install`, and rebuild the app.

## Goal

Allow external developers to install and run plugins without forking or rebuilding.

## Options to evaluate

### 1. Pre-bundled plugin files (most viable)
Plugins are distributed as self-contained `.js` bundles (like browser extensions).
The app has a plugin loader that `require()`s them at runtime from a local plugins
directory (e.g. `~/.canvaflow/plugins/`).

- Pros: no repo fork needed, works with any npm dep (bundled in)
- Cons: requires a plugin loader, security sandboxing (plugins run in the same
  Electron process), and a defined plugin API surface the loader can call into
- Prior art: VS Code extensions, Obsidian plugins

### 2. CDN-loaded dependencies
Plugins declare CDN URLs for their dependencies and load them at runtime via dynamic
`import()` or a script tag injector. The plugin code itself lives in the plugins
folder and is bundled normally.

- Pros: simple, no loader needed
- Cons: only works for browser-compatible libraries; breaks offline; no Node/Electron
  native packages; CDN availability is a runtime risk

### 3. VS Code–style extension host
A separate sandboxed process hosts plugins. The app exposes a stable extension API.
Plugins are distributed via a marketplace as `.vsix`-style archives.

- Pros: true isolation, rich ecosystem possible
- Cons: enormous scope; requires a full extension API design; overkill for current
  stage of the project

## Recommended path (when the time comes)

Start with **option 1**:

1. Define a `PluginManifest` interface (id, name, version, entry point)
2. At app startup, scan `~/.canvaflow/plugins/` for manifest files
3. Dynamically `require()` each plugin's bundle
4. Plugin bundles call a registration API (`canvaflow.registerPlugin(manifest)`)
   that contributes `SettingsSection` and `CanvasMount` components — same shape
   as the current v2 `Plugin` interface
5. Plugin authors bundle their code with their deps (esbuild / tsup) and distribute
   the output folder

The v2 `Plugin` interface (`id`, `SettingsSection`, `CanvasMount`) is already the
right contract — v3 just moves the registration from compile-time to runtime.

## Current state (v2)

- Plugin code: fully self-contained in `src/plugins/<name>/`
- Registration: one line in `src/plugins/index.ts`
- npm deps: must be added to root `package.json` + rebuild required
- Target audience: internal / repo contributors only
